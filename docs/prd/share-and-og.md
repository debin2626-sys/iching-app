# PRD：占卜结果页「分享这一卦」+ 动态 OG 图

> v2 — 基于创始人反馈修正 8 项

## 背景

当前占卜结果页（`/result`）的分享体验有两个问题：
1. 分享到微信/小红书/Twitter 时，预览图是通用的站点 OG 图，看不到卦名、卦象等个性化信息，点击欲低
2. 分享按钮用的是内联简化版（alert 提示），而项目里已有更完善的 `ShareButtons.tsx` 组件未被使用

这是修正版分析报告中「最高 ROI 的低垂果实」——把留存动作变成获客动作。

## 目标

- 用户完成占卜后，能一键分享结果页链接或保存分享图片
- 分享出去的链接在社交平台上显示个性化预览图（卦名 + 卦象 + 金句）
- 覆盖链接分享（微信/Twitter/Telegram）和图片分享（小红书/Instagram）两条路径
- 所有分享链接带 UTM 参数，GA4 可区分「直接访问」和「分享回流」
- 降低分享摩擦，提升自然传播率

## 非目标

- 不做微信 JS-SDK 深度集成（需要企业资质，等个体工商户执照下来再做）
- SEO 策略（`robots: { index: false }`）本期不变，待后续专项评估 — 这是暂缓决策，非已拍板决策。长远看结果页不索引会失去「雷天大壮解读」等 SEO 长尾流量，与获客策略有冲突，需要单独评估

## 功能拆解

### 功能 1：动态 OG 图

**触发条件：** 社交平台爬虫抓取 `/result?hexagram=1&...` 时自动生成

**OG 图内容（方案 A — 不含用户问题）：**
- 卦号 + 卦名（中文大字）+ 英文名
- 六爻图示（用 Unicode 符号 ⚊ ⚋ 表示阳爻阴爻）
- 一句金句（卦辞原文，如「大壮：利贞」）— 零 AI 味，自带古典权威感
- 右下角小字：51yijing.com（水印/回流锚 — 用户存图再发时链接会丢失，这是唯一的回流入口）

**为什么不放用户问题（question）：**
- question 是用户自由输入，几乎每人不同 → 缓存命中率接近 0%，30 天 CDN 缓存形同虚设
- 方案 A 缓存键 = `hexagram` + `locale`，64 卦 × 3 语言 = 192 张图，可全量预渲染，全站永远 CDN 命中
- 附加好处：OG 图变成「品牌资产」，每次分享看到的是同一张精心设计的卦图，比随机截断的用户输入（"我该不该分手..."）审美稳定得多

**变爻处理：** OG 图不展示变爻（变爻因人而异，会破坏缓存策略）。变爻信息在结果页内展示。

**尺寸与安全区：**
- 画布尺寸：1200×630（Facebook/Twitter/Telegram 标准）
- 微信朋友圈安全区：关键元素（卦名、卦象、金句）必须在中心 630×630 区域内，因为朋友圈分享会裁成方形预览
- 左右各 285px 为装饰区，可放边框/背景纹理，但不放关键信息

**设计规范：**
- 视觉风格：复用现有 OG 图设计语言（米白背景 `#f8f5f0`、双层金色边框 `#c4b08a`、四角装饰）
- 字体：Noto Serif SC Bold（复用 `loadNotoSerifSC`）
- 六爻颜色：与整体米白+金色调性协调，具体色值由 UX 评估（建议深朱红 `#8b2c2c` 或朱砂印色，避免纯红破坏整体美学）

**缓存策略：**
- HTTP 响应头：`Cache-Control: public, immutable, max-age=2592000`（30 天）
- 缓存键：`hexagram`（卦号）+ `locale`（语言），共 192 种组合
- Nginx 层：已有图片缓存配置（30 天），OG 图路径匹配后自动生效
- 可选优化：build 时预渲染 192 张图为静态文件，彻底消除运行时 Satori 开销

**Fallback 机制（三级降级）：**

| 级别 | 触发条件 | 降级行为 |
|------|----------|----------|
| L1 正常 | hexagram 有效（1-64）、字体和资源加载成功 | 完整个性化 OG 图（卦名+卦象+金句+水印） |
| L2 部分降级 | hexagram 有效但 locale 缺失，或字体加载失败 | 精简版（卦象+卦名+水印，系统 serif 字体） |
| L3 兜底 | hexagram 无效/缺失，或 Satori 渲染异常 | 返回静态站点通用 OG 图（`/og-image.png`） |

三级每一级都必须有明确的错误日志上报（console.error，标注降级级别和原因），不能静默失败。未来接 Sentry 后自动上报。

**技术方案：**
- 新建 `src/app/[locale]/result/opengraph-image.tsx`
- 从 URL query params 读取 `hexagram`（卦号）
- 通过卦号查询卦名+卦辞数据（内联 64 卦映射表，不查数据库）
- 用 Satori 渲染，复用现有 OG 图的边框/装饰/字体加载逻辑

### 功能 2：分享按钮升级 + UTM 追踪

**分享渠道优先级与平台适配：**

```
环境检测优先级：
1. 微信内置浏览器 → 引导浮层（详见下方微信环境特殊处理）
2. 支持 Web Share API 的移动端 → navigator.share()（覆盖 iOS Safari、Android Chrome）
3. 桌面端 → 复制链接到剪贴板
4. 所有环境 → 微博分享链接（已有实现）
```

**UTM 参数（本期必须，不是后续扩展）：**
- 所有分享出去的链接统一追加：`?utm_source=share&utm_medium=hexagram&utm_content={卦名拼音}`
- 没有 UTM，上线后无法区分「直接访问」和「分享回流」，GA4 里一锅粥，无法证明分享功能有效
- GA4 事件埋点（`share_click` 等）可以放 v2，但 UTM 参数必须 v1
- 工期增量：30 分钟，和分享链接生成合并即可

**微信环境特殊处理：**
- 检测 `navigator.userAgent` 包含 `MicroMessenger`
- 显示引导浮层：
  - 内容：「点击右上角 ··· 分享给朋友」+ 箭头指向右上角（微信 H5 行业标准）
  - 关闭方式：3 秒自动消失 + 可手动点击关闭
  - 频率控制：同一用户同会话只弹第一次，后续点击不再弹出（sessionStorage 记录）
- 这是过渡方案，等个体工商户执照 + 服务号资质下来后再接 JS-SDK

**按钮位置：** 保持在结果页底部操作栏，与「再次占卜」「返回首页」并列

**提示方式：** 用 `useToast` 替代 `alert`

**分享文案模板：** `「{卦名}」— {金句} | 51yijing.com`

### 功能 3：保存为图片

**为什么必须有：** 小红书和 Instagram 不接受链接分享，只能发图。没有「保存图片」，小红书这条获客路径等于断了。

**实现方案：**
- 按钮文案：「保存卦图」
- 点击后在客户端生成分享图片（PNG）
- 图片内容 — 竖版，不放 AI 摘要：
  - 卦名（大字）+ 英文名
  - 六爻图示
  - 一句金句（卦辞/爻辞原文，如「大壮：利贞」）— 8 字点题，零 AI 味，自带古典权威感
  - 51yijing.com 水印
  - 二维码预留位（本期可先放文字短链，二维码等短链服务就绪后加）
- 移动端：触发下载到相册
- 桌面端：触发下载 PNG 文件

**为什么不放 AI 解读摘要：**
1. 心理学问题：用户在朋友圈看到「AI 解读」四个字，神秘感/权威感 -50%。国学产品的气质和「AI 分析」是冲突的
2. 钩子逻辑问题：如果摘要已经写在图上，用户就没理由点链接回站了，保存图片转发反而断了回流
3. 正确做法：图片只给「钩子」（卦名+金句），让看到的人想知道答案必须回站

**尺寸：**
- 主尺寸：1242×1656（3:4，小红书原生推荐）— 小红书优先级高于 Instagram
- 备选：后续可加 1080×1350（4:5，Instagram feed 推荐），本期先保一个尺寸

**与 OG 图的区别：**

| | OG 图 | 分享图片 |
|---|---|---|
| 触发方式 | 爬虫自动抓取 | 用户主动点击保存 |
| 尺寸 | 1200×630（横版） | 1242×1656（竖版） |
| 生成位置 | 服务端（Satori） | 客户端（html2canvas/canvas） |
| 内容 | 卦名+卦象+金句+水印 | 卦名+卦象+金句+水印+回流入口 |
| 用途 | 微信/Twitter 链接预览 | 小红书/Instagram 图片分享 |

## 多语言

- OG 图 & 分享图片：根据 `[locale]` 路由参数决定显示中文/英文卦名和金句
- 金句来源：中文用卦辞原文，英文用对应英译
- 分享文案：走 `next-intl` 现有 i18n 体系
- 支持 zh / zh-TW / en 三种语言

## 数据依赖

| 数据 | 来源 | 说明 |
|------|------|------|
| 卦号 | URL query `hexagram` | 1-64 |
| 卦名（中/英） | 内联 64 卦映射表 | 已有 `HexagramData` 类型，提取 nameZh/nameEn/judgmentZh/judgmentEn 子集 |
| 卦辞/金句 | 同上映射表 | 中文：judgmentZh 原文；英文：judgmentEn |
| 六爻图示 | 卦号 → 上下卦三位 → Unicode ⚊/⚋ | 现有 `upperTrigram`/`lowerTrigram` 数据可用 |
| locale | URL path `[locale]` | zh/zh-TW/en，缺失时 fallback 到 zh |
| 多语言 fallback 链 | next-intl 配置 | zh-TW → zh → en |

64 卦映射表数据源：现有 `/api/hexagram/{number}` 返回的数据结构，提取为静态 JSON 内联到 OG 图生成代码中（不运行时查数据库）。

## 验收标准

### OG 图
1. 微信/Twitter/Telegram 中粘贴结果页 URL，预览图显示卦名+卦象+金句（非通用 OG 图）
2. 关键元素在微信朋友圈方形裁切后仍完整可见（中心 630×630 安全区）
3. 响应头包含 `Cache-Control: public, immutable, max-age=2592000`
4. 同一卦号+同一语言的 OG 图第二次请求走缓存（验证 Nginx 缓存命中）
5. hexagram 参数缺失/无效时返回通用 OG 图（L3 降级），不报错
6. 字体加载失败时降级到系统字体（L2 降级），不出空白图
7. 每级降级都有 console.error 日志，标注级别和原因

### 分享按钮
8. 移动端（非微信）点击分享按钮，弹出系统分享面板
9. 微信内点击分享按钮，显示引导浮层（箭头+文案），3 秒自动消失，可手动关闭，同会话只弹一次
10. 桌面端点击分享按钮，链接复制到剪贴板，toast 提示「已复制」
11. 所有分享链接包含 UTM 参数（`utm_source=share&utm_medium=hexagram&utm_content={卦名}`）
12. GA4 中能按 utm_source=share 筛选出分享回流流量

### 保存图片
13. 点击「保存卦图」，生成竖版分享图片并触发下载
14. 图片包含卦名、六爻图、金句（卦辞原文）、51yijing.com 水印
15. 图片不包含 AI 解读摘要
16. 图片在小红书发布时比例正确（3:4），关键信息不被裁切

### 通用
17. 中/英文/繁体三种语言下 OG 图、分享文案、分享图片均正确
18. 所有降级场景有错误日志，不静默失败
